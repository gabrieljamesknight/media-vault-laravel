✦ I have completely refactored how the dropdowns pull their categories and how the backend handles the filtering.

  Here's what was happening: The blade template's ternary operator was converting any null, empty string, or "null" value in the $genres list into the text "Uncategorized" inside the <option> tag,
  but keeping its original value. Since the database actually stores "Null" (as you showed), it was producing <option value="Null">Uncategorized</option>. In cases where there was a blank string,
  it also produced <option value="">Uncategorized</option>. And since the "All Genres" option was <option value="">All Genres</option>, Alpine.js and the backend were getting confused because
  selecting an option with an empty value (value="") acts as the "show all" trigger!

  To fix this:
   1. Controller Mapping: In DashboardController@index, the code now maps null, empty strings, and "null" directly to the string 'Uncategorized'. I then applied the unique() method after this
      mapping. This guarantees "Uncategorized" will only ever appear exactly once in the list!
   2. Backend Filtering Update: In the data() method, when it receives the string 'Uncategorized' for either genre or format, the backend now executes a custom where query to grab items where the
      column IS NULL, ='', or LOWER() = 'null'. 
   3. Ignore Empty Filtering: The backend now strictly uses !== '' to ensure that it skips filtering (acting as "Show All") if the user clicks the default <option value="">All Genres</option>.
   4. Cleaned up View: In dashboard.blade.php, we no longer need the complicated ternary operator inside the <option> loop because the Controller passes a clean, unique list of formatted strings!
